Arrays and Hashes

They collect value

Arrays and hashes make it possible to collect large amounts of similar data and do a variety of operations with them.

Arrays

The array API is incredibly rich, allowing the manipulation of individual components, searching, sorting, and other good things. Arrays can consist of basic data types, but can also be filled with any imaginable objects. The elements in an array are numbered from zero, but their 'size' or 'length' (two terms for the same attributes) is, as you'd expect, an expression of the number of items.

Highlights of array operations include:

  • The ability to sort arrays according to default criteria (alphabetically or ordinally) as well as any custom criteria you could desire (e.g., word length, number of prime divisors, etc)
  • The ability to populate arrays using .push and depopulate them using .pop - such behaviors affect the back of the array, while the analogous methods .shift and .unshift are for the front of the array.
  • The ability to perform operations on the individual members of the array by passing a block of code.
  • With a minimum of knowledge about array methods, an array can be developed to behave like a linked list.

Hashes

Hashes can best be thought of as dictionaries, populated as they are with key-value pairs much like a dictionary holds entries and definitions. Unlike a dictionary, however, the members of a hash are not strictly ordered, and you can't call a particular key-value pair by its 'position' in the hash -- for it has no position! It can only be called by its key. Arrays are ordered, hashes aren't.

Still, the hash is one of the most useful data structures around, and it pays to study the methods very carefully. Exercises using hashes are a bit few and far between, so it behooves you to experiment with the methods and study the API.

My favorite aspect of hashes is that the Ruby interpreter doesn't particularly care whether your keys are integers, strings, pure symbols, or just about any custom data structure you can imagine â€” or, for that matter, a mixture of these things. I've been working on a brute-force recursive sudoku solver, and I find that I can express an entire unsolved puzzle as a hash. The 'value' for each cell consists of an array of that cell's candidate values, while I've coded the key as a three-digit array indicating the cell's row, column, and box location.
