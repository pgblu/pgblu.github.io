Ruby Hashes vs. JavaScript Objects

Let's discuss values

On first glance, Ruby hashes and objects in JS have obvious similarities. Both are structured as dictionaries, with the option of keys being treated symbolically, and it is easy in both cases to access (get) or mutate (set) the values.

In Ruby, hashes are but one of many kinds of core objects. Along with arrays, they are the jewels in the Ruby crown, with an incredibly rich API. Ruby makes no assumptions about the value types of hashes. They can be strings, symbols, integers, arrays, even other hashes -- but the real power lies in the fact that they can be populated by custom objects, or even a different object type for each key.

Ruby's hashes have the methods of the 'Enumerable' module mixed in, which is another source of tremendous flexibility. It permits the keys of the array to be accessed through a single command, leading to traversing, chunking, mapping, and sorting according to custom criteria.

Granted, all of these things can be done with JS objects -- and then some! -- but that is not really the point. The point is that a hash makes no assumptions about how it will be used. JS objects, on the other hand, are assumed to be the models of real-world objects, and the JS functions built into a given application are meant to act upon them to derive new data.

I'm tempted to put it this way: hashes are a means to an end, while JS objects are points of departure, forming the backbone of your application.
